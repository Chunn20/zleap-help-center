# Zleap 独立帮助中心 0-1 建设与内容管理 PRD

## 1. 项目概述

Zleap 帮助中心是从 0 到 1 搭建的独立用户自助服务平台。项目启动前，产品帮助内容主要沉淀在飞书文档中，适合内部协作，但不适合作为正式对外帮助中心：页面层级复杂、视觉不统一、检索效率低、移动端阅读体验弱，也难以体现产品品牌和专业度。

本项目的核心目标，是将原本分散在飞书中的帮助内容，重组为面向用户的独立帮助中心。项目基于 Fumadocs / Next.js 搭建，覆盖首页、搜索结果页、文档详情页、产品支持、企业版介绍、更新日志前台展示和更新日志后台管理等模块。

项目价值不只是“把文档搬到网页上”，而是通过信息架构、页面框架、搜索体验、内容规范、移动端适配和后台维护流程，帮助用户更快理解产品、查找教程、解决问题，同时降低内部产品 / 运营人员维护帮助内容和更新日志的成本。

## 2. 项目背景

项目早期，Zleap 的帮助说明主要以飞书文档形式维护。随着产品功能增加，飞书文档逐渐暴露出几个问题：

- 内容分散在多个文档中，缺少统一的信息架构，用户难以判断从哪里开始阅读。
- 页面结构复杂，视觉样式依赖编辑者手动维护，对外展示不够美观和专业。
- 飞书文档的搜索、跳转和浏览体验有限，用户查找具体操作教程的路径较长。
- 移动端阅读多列内容、长文档和图片说明时体验不稳定。
- 更新日志依赖人工整理和手动修改数据文件，容易出现格式错误。

因此，本项目从“内部文档沉淀”升级为“独立帮助中心建设”，先完成核心站点与文档体系搭建，再持续优化搜索、页面样式、移动端和内容维护后台。

## 3. 目标用户

| 用户类型 | 使用场景 | 核心诉求 |
| --- | --- | --- |
| 新用户 | 首次访问帮助中心，了解 Zleap 是什么、如何下载、如何快速开始、如何创建信息源 | 快速建立产品认知，按步骤完成基础配置 |
| 存量用户 | 遇到具体操作问题，通过搜索或目录定位教程，例如信息源、通知中心、语言设置、更新日志 | 更短路径找到答案，返回搜索结果时保留上下文 |
| 内部运营 / 产品人员 | 维护帮助文档、补充版本更新记录、发布更新日志 | 降低内容维护门槛，减少手动修改 JSON 的出错概率 |

## 4. 核心问题与需求目标

| 问题 | 用户影响 | 对应需求目标 | 当前项目中的实现方式 |

## 5. 需求范围

### 5.1 本期已实现范围

- 从 0 搭建基于 Fumadocs / Next.js 的 Zleap 独立帮助中心。
- 设计帮助中心首页，提供搜索入口、核心入口卡片、常用问题和更新日志入口。
- 建立文档信息架构，覆盖产品介绍、新手入门、信息管理、产品支持、联系反馈等模块。
- 建设文档详情页阅读框架，包括左侧目录、正文内容、提示框、图片和页面目录。
- 搭建搜索结果页，并实现搜索结果浏览位置记忆。
- 隐藏文档详情页顶部搜索栏，减少阅读干扰。
- 统一 Lightbulb 提示框样式，并提供 `fix_tip_boxes.py` 批量处理脚本。
- 统一顶部导航标题为「智跃帮助中心」，并统一导航菜单间距。
- 统一所有页面标题间距标准，覆盖 28 个文件。
- 完成首页三卡片布局优化。
- 完成企业版页面结构重构和标题间距优化。
- 完成信息管理、企业版、文档正文等多个页面的移动端响应式适配。
- 修复文档上传页面卡片跳转链接。
- 更新小红书相关入口 / 图标内容。
- 建设更新日志前台展示页。
- 建设更新日志轻量后台：访问 `/admin/changelog`，输入密码后发布新版本记录。
- 更新日志数据通过 `data/changelog.json` 持久化，前台更新日志页动态读取。
- 补充历史版本记录：当前 `data/changelog.json` 包含 `V1.8.0` 至 `V0.10.0` 共 20 条记录；其中 `V1.0.0` 至 `V0.10.0` 代码可数为 11 条，“共 12 个”待进一步确认。
- 修复 JSON 格式问题，当前 `data/changelog.json` 可正常解析。

### 5.2 本期不包含范围

- 不包含用户行为数据埋点分析。
- 不包含复杂权限系统。
- 不包含多角色后台审批流。
- 不包含在线富文本编辑器。
- 不包含后台版本记录编辑、删除能力：当前代码和使用说明均显示仅支持新增发布，编辑 / 删除为待确认或后续能力。
- 不包含后台实时预览：当前代码可发布后跳转前台查看，未确认实时预览模块。

## 6. 功能需求详述

### 6.1 帮助中心首页建设

#### 需求说明

建设独立帮助中心首页，替代飞书文档入口，让用户进入后能快速理解帮助中心提供哪些内容，并进入常用功能文档。

#### 使用场景

新用户首次访问帮助中心，希望快速了解 Zleap 的核心功能、上手路径和常用教程入口。

#### 交互逻辑

- 首页顶部展示品牌导航和搜索入口。
- Hero 区承载产品帮助中心定位。
- 三个核心入口卡片引导用户进入快速开始、信息管理、产品介绍 / 支持等文档。
- 常用搜索标签和快捷入口减少用户查找成本。

#### 页面 / 路由 / 文件依据

- 路由：`/`
- 文件：`app/page.tsx`
- 资源：`public/background.png`、`public/images/home/1.png`、`public/images/home/2.png`、`public/images/home/3.png`

#### 验收标准

- 首页可清晰展示搜索入口和核心文档入口。
- 三卡片文字与图片不重叠。
- 移动端不出现横向溢出。

#### 产品价值

将原本飞书文档里的内容入口前置为产品化首页，降低新用户首次查找成本。

### 6.2 信息架构与文档体系建设

#### 需求说明

将飞书文档中的说明内容重组为结构化帮助中心目录，按用户任务和产品模块组织文档。

#### 使用场景

用户希望通过目录逐层查找某个功能说明，例如信息源创建、信息搜索、通知中心或企业版能力。

#### 交互逻辑

- 通过 `meta.json` 配置文档分组和展示顺序。
- 文档详情页展示侧边栏目录。
- 支持按产品介绍、新手入门、信息管理、产品支持、联系反馈组织内容。

#### 页面 / 路由 / 文件依据

- 文件：`content/docs/meta.json`
- 文件：`content/docs/about-zleap/meta.json`
- 文件：`content/docs/getting-started/meta.json`
- 文件：`content/docs/information-management/meta.json`
- 文件：`content/docs/support/meta.json`
- 文件：`content/docs/contact/meta.json`

#### 验收标准

- 用户可以从总目录进入主要文档模块。
- 文档目录顺序符合用户理解产品和完成任务的路径。
- 新增文档可通过目录配置接入帮助中心。

#### 产品价值

把飞书文档的线性内容整理成可导航的信息架构，提高用户自助查找效率。

### 6.3 搜索结果页浏览位置记忆

#### 需求说明

用户在搜索结果页浏览多个结果时，从某一条结果进入文档详情页，再返回搜索结果页，应恢复到离开前的滚动位置。

#### 使用场景

用户搜索“信息源”“GitLab”“更新日志”等关键词，连续查看多条搜索结果。

#### 交互逻辑

- 用户在搜索结果页滚动时，页面将 `window.scrollY` 写入 `sessionStorage`。
- 用户返回搜索页后，页面读取 `search-scroll-position`。
- 为避免结果尚未加载完成导致恢复失败，页面通过多个延迟时间点重复执行 `window.scrollTo`。

#### 页面 / 路由 / 文件依据

- 路由：`/search?q=关键词`
- 文件：`app/search/page.tsx`

#### 验收标准

- 用户从搜索结果页进入任一详情页后返回，页面恢复到离开前位置。
- 搜索结果重新加载后仍能恢复滚动位置。
- 无搜索词时不展示旧结果。

#### 产品价值

降低连续查找过程中的重复操作成本，让搜索页更像可持续浏览的信息列表。

### 6.4 文档详情页阅读焦点优化

#### 需求说明

文档详情页隐藏顶部搜索栏，仅保留帮助中心品牌和导航结构，减少正文阅读干扰。

#### 使用场景

用户进入教程文档后，需要按步骤阅读、对照截图完成操作。

#### 交互逻辑

- 文档布局统一使用 `HelpCenterHeader hideSearch`。
- 搜索入口仍保留在首页和搜索页等查找场景中。

#### 页面 / 路由 / 文件依据

- 路由：`/docs/**`
- 文件：`app/docs/layout.tsx`
- 文件：`app/components/HelpCenterHeader.tsx`

#### 验收标准

- `/docs/**` 文档详情页顶部不展示搜索输入框。
- 首页和搜索页仍可正常搜索。

#### 产品价值

按场景区分“查找”和“阅读”任务，帮助用户在详情页聚焦内容理解。

### 6.5 页面样式规范与提示框统一

#### 需求说明

建立帮助中心页面样式规范，统一标题间距、顶部导航、Lightbulb 提示框、卡片和移动端样式。

#### 使用场景

用户在不同文档页面间切换时，获得一致的阅读体验；内部维护人员新增文档时有统一格式参考。

#### 交互逻辑

- 统一 H1/H2/H3 标题间距。
- 顶部导航标题统一为「智跃帮助中心」。
- 多处 Lightbulb 提示框统一为相同背景、间距、圆角和图标样式。
- 使用 `fix_tip_boxes.py` 批量处理历史文档提示框。

#### 页面 / 路由 / 文件依据

- 文件：`app/components/HelpCenterHeader.tsx`
- 文件：`app/global.css`
- 文件：`fix_tip_boxes.py`
- Git 记录：统一标题间距覆盖 28 个文件；`5cf2ac0` 涉及提示框样式统一和移动端卡片适配。

#### 验收标准

- 页面标题间距在主要文档中保持一致。
- 顶部导航统一展示「智跃帮助中心」。
- Lightbulb 提示框具有统一背景、内边距和圆角。
- 提示框在移动端不横向溢出。

#### 产品价值

将飞书文档式的非统一排版，升级为可长期维护的帮助中心页面规范。

### 6.6 移动端响应式适配

#### 需求说明

针对移动端调整文档字号、标题层级、图片、代码块、目录和多列卡片布局，保证小屏可读、可点、可导航。

#### 使用场景

用户在手机上查看帮助中心，按教程操作 App 或网页产品。

#### 交互逻辑

- 文档正文在 `max-width: 768px` 下调整 H1/H2/H3、段落、列表、图片、代码块样式。
- 信息管理入口卡片在移动端由 4 列调整为 2 列。
- 企业版三列内容在移动端调整为 1 列。
- 文档侧边栏和本页目录在移动端保持可访问。

#### 页面 / 路由 / 文件依据

- 文件：`app/global.css`
- 文件：`content/docs/information-management/information-management.mdx`
- 文件：`content/docs/about-zleap/enterprise.mdx`

#### 验收标准

- 移动端卡片不出现横向溢出。
- 文档标题、正文、图片、代码块在小屏下可读。
- 移动端可以访问总目录和单页目录。

#### 产品价值

覆盖用户边操作边查阅的移动场景，降低小屏阅读压力。

### 6.7 企业版页面建设与重构

#### 需求说明

企业版页面从普通说明文档升级为更清晰的产品介绍页面，突出企业管理挑战、Zleap 解决方案、核心功能、应用场景和版本对比。

#### 使用场景

潜在企业客户或内部销售 / 产品人员需要快速理解企业版定位、核心价值和适用场景。

#### 交互逻辑

- 页面先给出企业版概览和核心定位。
- 再通过三张挑战卡片解释问题。
- 对应展示三张解决方案卡片。
- 后续补充核心功能、技术优势、应用场景、版本对比和硬件配置。

#### 页面 / 路由 / 文件依据

- 路由：`/docs/about-zleap/enterprise`
- 文件：`content/docs/about-zleap/enterprise.mdx`
- 图片：`public/images/about-zleap/enterprise-overview.png`、`public/images/about-zleap/enterprise-solution.png`

#### 验收标准

- 页面存在独立的挑战模块和解决方案模块。
- 三列卡片在桌面端层级清晰，在移动端自动转为单列。
- 图标背景、标题和正文层级可区分。

#### 产品价值

将企业版从功能罗列改为“问题 - 方案 - 价值 - 场景”的表达结构，更适合对外展示和销售沟通。

### 6.8 更新日志后台管理系统

#### 需求说明

提供轻量更新日志管理后台，内部人员通过页面录入版本号、日期、新增功能和优化内容，发布后写入 `data/changelog.json`，前台更新日志页动态展示。

#### 使用场景

产品或运营人员在版本发布后，快速补充更新日志，减少手动修改前端页面或 JSON 文件的频率。

#### 交互逻辑

- 访问 `/admin/changelog`。
- 输入管理员密码；默认密码由代码确认为 `zleap2024`，也可通过环境变量 `CHANGELOG_ADMIN_PASSWORD` 覆盖。
- 粘贴固定格式文本，系统解析版本号、日期、新增项和优化项。
- 点击发布后，请求 `/api/changelog`，新记录写入 `data/changelog.json` 首位。
- 发布成功后跳转 `/docs/support/changelog` 查看展示结果。
- 管理员账号 `admin`：待确认，当前代码仅校验密码，未发现账号字段。
- 编辑、删除、实时预览：当前代码未确认，属于后续扩展能力。

#### 页面 / 路由 / 文件依据

- 后台路由：`/admin/changelog`
- 前台路由：`/docs/support/changelog`
- 文件：`app/admin/changelog/page.tsx`
- 文件：`app/api/changelog/route.ts`
- 文件：`app/api/changelog/auth/route.ts`
- 文件：`app/components/ChangelogDynamic.tsx`
- 文件：`更新日志管理系统使用说明.md`

#### 验收标准

- 输入正确密码后可进入发布页面。
- 按格式输入内容后可发布新版本。
- 新版本写入 `data/changelog.json` 首位。
- 前台更新日志页可读取并展示新版本。
- 格式错误时页面给出错误提示。

#### 产品价值

把更新日志维护从代码修改转为轻量发布流程，降低内部维护门槛和格式出错概率。

## 8. 页面原型说明

### 8.1 帮助中心首页

- 页面目标：作为帮助中心入口，帮助用户快速搜索问题或进入核心文档。
- 核心模块：顶部导航、Hero 搜索框、搜索建议 / 历史记录、常用搜索标签、认识 Zleap 三卡片、使用 Zleap 快捷入口。
- 用户操作路径：进入首页 -> 输入关键词或点击标签 -> 进入搜索结果页 / 详情页；也可直接点击快捷入口进入文档。
- 关键状态：无输入、输入中、展示建议、展示搜索历史、清空搜索。
- 原型交互点：搜索输入、清空按钮、搜索按钮、建议下拉、卡片 hover、快捷入口跳转。

### 8.2 搜索结果页

- 页面目标：集中展示搜索结果，并支持用户连续浏览。
- 核心模块：顶部导航搜索框、搜索框、结果统计、结果列表、空状态、加载状态。
- 用户操作路径：输入关键词 -> 查看结果 -> 点击结果进入文档 -> 返回结果页并恢复位置。
- 关键状态：加载中、无结果、有结果、返回恢复滚动位置。
- 原型交互点：关键词高亮、面包屑路径、结果标题跳转、正文摘要跳转、滚动位置记忆。

### 8.3 文档详情页

- 页面目标：提供聚焦的帮助文档阅读和步骤指导。
- 核心模块：顶部品牌导航、左侧总目录、正文、右侧 / 弹出式本页目录、图片、提示框、跳转卡片。
- 用户操作路径：从首页、搜索结果或侧边栏进入文档 -> 阅读内容 -> 使用目录跳转章节 -> 点击图片或链接继续操作。
- 关键状态：桌面端双目录结构、移动端目录可访问、提示框统一、顶部搜索隐藏。
- 原型交互点：侧边栏折叠、目录跳转、图片查看、提示框样式、移动端布局切换。

### 8.4 更新日志后台页

- 页面目标：帮助内部人员发布更新日志。
- 核心模块：密码登录、内容输入区、格式示例、发布按钮、取消按钮、成功 / 错误提示。
- 用户操作路径：访问 `/admin/changelog` -> 输入密码 -> 粘贴版本内容 -> 发布 -> 跳转前台查看。
- 关键状态：未登录、密码错误、内容为空、发布中、发布成功、格式错误。
- 原型交互点：密码提交、文本解析、发布按钮禁用态、错误提示、发布成功跳转。

## 9. 验收标准

- 帮助中心首页可清晰承载搜索入口、核心文档入口和常用问题入口。
- 文档目录结构覆盖产品介绍、新手入门、信息管理、产品支持和联系反馈。
- 用户从搜索结果页进入详情页并返回后，页面恢复到原滚动位置。
- 文档详情页不展示顶部搜索栏。
- 首页三卡片和快捷入口在桌面端布局稳定，移动端不横向溢出。
- Lightbulb 提示框背景、内边距、圆角和图标表现统一。
- 顶部导航统一展示「智跃帮助中心」。
- 所有页面标题间距标准统一。
- 移动端文档标题、正文、图片、代码块和目录可正常阅读与操作。
- 企业版页面拆分为挑战和解决方案模块，并在移动端单列展示。
- 文档上传页面卡片跳转链接正确。
- 更新日志后台可以新增版本记录。
- `data/changelog.json` 更新后，前台 `/docs/support/changelog` 能正常展示。
- JSON 格式错误不再导致页面渲染失败。

## 11. 风险与后续优化

- 缺少搜索词、点击率、停留时长等行为数据，后续难以量化判断帮助文档是否真正降低查找成本。
- 更新日志后台目前是轻量发布能力，后续可补充账号体系、权限控制和操作记录。
- 后台仅确认支持新增发布，编辑、删除和实时预览需要进一步产品化设计。
- 帮助文档内容仍可继续从“功能说明”转向“用户任务说明”，减少用户理解成本。
- 后续可增加无结果搜索词分析，用于反向指导文档补充。
- 当前部分文档样式仍直接写在 MDX 中，后续可沉淀为统一组件，降低长期维护成本。

## 12. 产品复盘

帮助中心不是产品附属说明，而是用户自助理解产品的重要入口。本项目的关键价值，是把原本散落在飞书文档中的内容，从内部协作文档升级为独立、可搜索、可导航、可维护的用户帮助中心。

从产品经理视角看，这个项目不是简单修补页面，而是完整经历了从问题识别到 0-1 建设的过程：识别飞书文档作为外部帮助中心的局限，重新拆解目标用户和信息架构，设计首页、搜索、文档详情和更新日志后台等关键路径，并在上线后持续优化标题间距、导航、提示框、移动端和跳转问题。

这个项目体现了我从真实业务问题出发，完成需求拆解、信息架构设计、交互路径设计、内容维护流程建设和上线复盘沉淀的能力。
